# Repository Guidelines

## Project Structure & Module Organization
- `init.lua` is the entry point; it loads `config.settings`, `config.keymaps`, `config.plugins`, and `config.autocmds`.
- Core behavior lives in `lua/config/`; feature plugins are split under `lua/plugins/` (one spec table per file).
- Shared helpers stay under `lua/utils/` (borders, keymap helpers, LSP tweaks) to keep configs declarative.
- Snippets belong to `lua/snippets/`; update the matching page in `docs/` whenever you add or rename one.
- `lazy-lock.json` is generated by Lazy.nvim; only update it via Lazy commands, never by hand.

## Build, Test, and Development Commands
- `nvim --headless "+Lazy sync" +qa` installs or updates plugins exactly as pinned in `lazy-lock.json`.
- `nvim --headless "+Lazy check" +qa` audits plugin health and reports missing dependencies.
- Inside Neovim run `:checkhealth` after changing providers, and `:Mason` to confirm language servers.
- Format Lua with `stylua --indent-type Tabs --indent-width 2 lua` (Stylua 0.20+; no repo-level config is committed).
- When adjusting formatters, use `:ConformInfo` to verify the active tool per filetype.

## Coding Style & Naming Conventions
- Use hard tabs for indentation, aligning with the existing Blink/Conform configs; trim trailing whitespace.
- Module filenames follow `snake_case.lua`; exported tables use descriptive keys such as `opts`, `dependencies`, and `keys`.
- Favor declarative plugin specs: return a list, avoid inline side effects beyond `config` functions.
- Local helper functions belong in `lua/utils/`; only keep them inside plugin spec files when they are truly plugin-specific.

## Testing Guidelines
- No automated suite exists; rely on headless checks plus manual verification in Neovim.
- **CRITICAL: Always test configuration changes in headless mode BEFORE claiming they work.**
  - Use `nvim --headless -c "e <test-file>" -c "sleep 3" -c "messages" -c "quit" 2>&1` to check for errors
  - Test LSP attachment: `nvim --headless -c "e lua/test.lua" -c "sleep 5" -c "lua local clients = vim.lsp.get_clients({bufnr=0}); print('LSP clients:', #clients); for _, c in ipairs(clients) do print('  - ' .. c.name) end" -c "quit" 2>&1`
  - Verify `:LspInfo` exists: `nvim --headless -c "e lua/test.lua" -c "sleep 3" -c "lua print('LspInfo:', vim.fn.exists(':LspInfo') == 2 and 'EXISTS' or 'MISSING')" -c "quit" 2>&1`
  - Clear caches if issues persist: `rm -rf ~/.local/state/nvim/lazy/* ~/.cache/nvim/*`
- After LSP or completion changes, open representative buffers (`*.lua`, `*.ts`, `*.heex`) and confirm diagnostics and completion still fire.
- Validate snippet changes with `:LuaSnipEdit` and update `docs/snippets.md` so contributors know the new shape.
- **Common pitfalls to avoid:**
  - Plugin loading order: Dependencies must load before they're required (use `dependencies = {}` in plugin specs)
  - API changes: Always check if plugin APIs have changed (e.g., `mason-lspconfig` v2.0.0 removed `setup_handlers()`)
  - Filetype restrictions: LSP servers should specify `filetypes = {}` to prevent unwanted attachment

## Commit & Pull Request Guidelines
- Follow Conventional Commits as seen in history (`feat:`, `fix:`, `chore:`); use imperative mood and keep subjects under 72 chars.
- Before opening a PR, run `stylua` and the headless Lazy checks, then note the results in the description.
- Summaries should outline the affected modules (`lua/plugins/lsp.lua`, etc.), the user-facing impact, and any follow-up tasks.
- Attach screenshots or terminal captures when altering UI components (statusline, colorscheme, dashboards) and link related issues or TODOs.
